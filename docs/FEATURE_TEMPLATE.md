# [기능 템플릿] 기능명 적기

이 템플릿은 새로운 기능 개발에 착수하기 전, 학습 목적에 맞추어 기획서 작성을 완료하는 용도로 사용합니다. 이 문서를 복사하여 `docs/features/` 폴더 또는 해당 위치에 별도 파일로 기입한 뒤 구현을 시작하십시오.

---

## 1. 기능 목적
* 이 기능이 해결하고자 하는 사용자 요구사항이나 학습 목표를 서술합니다.

## 2. 사용자 행동 (Trigger)
* 사용자가 브라우저에서 어떤 동작(예: 클릭, 타이핑 등)을 실행하는지 세부 기입합니다.
* 예: "사용자가 사이드바 하단 '새 페이지' 버튼을 마우스로 클릭한다."

## 3. 예상 동작 (Result)
* 사용자 행동의 결과로 화면이 어떻게 바뀌는지 기입합니다.
* 예: "비어 있는 상세 에디터 화면이 나오고, 사이드바 목록 맨 마지막에 '제목 없음' 문서가 나타난다."

## 4. 데이터 흐름 (Data Flow)
* 아래 다이어그램 형식을 이용해 데이터가 전달되고 갱신되는 상세 프로세스를 명세합니다.
```text
[사용자 행동]
              ↓
[이벤트 처리]
              ↓
[상태 변경]
              ↓
[변경 감지 / useEffect]
              ↓
[API 호출]
              ↓
[백엔드 요청]
              ↓
[응답 처리 / 상태 저장]
              ↓
[Props 전달 및 재렌더링]
```

## 5. 관련 상태 (React States)
* 이 기능에 직접적으로 관여하는 `useState` 목록과 그 상태의 위치(컴포넌트명)를 정의합니다.
* 예: `selectedPageId` (WorkspacePage)

## 6. 관련 컴포넌트
* UI 렌더링에 참여하는 컴포넌트와 그 역할을 나열합니다.

## 7. 관련 API (HTTP Contract)
* 호출하게 될 HTTP 메서드와 엔드포인트를 기입합니다.
* 예: `POST /api/pages`

## 8. 생성할 파일 [NEW]
* 새로 생성해야 하는 파일의 경로와 그 목적을 적습니다.

## 9. 수정할 파일 [MODIFY]
* 기존 파일 중 소스 코드를 추가/수정해야 하는 대상을 명시합니다.

## 10. 세부 구현 순서
* 어떤 파일부터 시작해서 어떤 순서로 조립해 나갈지 단계별로 명세합니다.
1. `types/` 파일 타입 선언
2. `api/` 통신 함수 구현
3. ...

## 11. useEffect 사용 여부와 이유
* 이 기능 구현에 `useEffect`를 사용하는지 적고, 사용하는 이유 혹은 사용하지 않고 이벤트 핸들러로 해결하는 이유를 명확히 작성합니다.

## 12. 로딩 및 오류 처리
