## 1. 아키텍처 및 폴더 구조 이해

### [ ] 1. Page와 Component의 차이를 명확히 구분하여 말할 수 있다.
* **스스로 답해보기**: 둘은 어떤 폴더에 분리되며, 왜 직접 UI 마크업을 Page 안에 다 쑤셔 넣지 않고 Component로 조립해서 그릴까요?
* **정답 힌트**: Page는 URL 라우팅 경로와 매핑되며 여러 컴포넌트를 조립하고 전체 비즈니스 상태를 분출하는 허브 역할을 하고, Component는 단일 책임 하에 재사용 가능한 뷰나 작은 로직을 캡슐화합니다.

### [ ] 2. Layout의 필요성과 `children` 프로퍼티의 역할을 설명할 수 있다.
* **스스로 답해보기**: 여러 화면에서 헤더와 사이드바가 그대로 유지되면서 본문만 교체될 때 Layout 컴포넌트는 어떻게 작동합니까?

## 2. 상태 관리 및 Props

### [x] 3. 상태(State)를 부모 컴포넌트에 선언해야 할 때와 자식 컴포넌트에 선언해야 할 때의 판단 기준을 설명할 수 있다.
* **스스로 답해보기**: 문서 선택 ID(`selectedPageId`)는 왜 개별 `PageListItem`이나 `Sidebar`가 아니라 `WorkspacePage`에 선언해야 했나요?
* **정답 힌트**: 상태를 서로 공유해야 하는 여러 자식 컴포넌트가 존재할 경우, 그들의 '가장 가까운 공통 조상(부모) 컴포넌트'로 상태를 끌어올려야(State Lifting) 데이터 흐름이 정상 작동합니다.

### [x] 4. Props가 아래로만 내려가는 흐름(단방향 데이터 흐름)을 코드로 짚어가며 추적할 수 있다.
* **스스로 답해보기**: 부모에서 선언한 상태 변경 함수(`setSelectedPageId`)는 어떻게 자식 컴포넌트 내부에서 마우스 클릭 시점에 호출될 수 있는지 설명하십시오.
* **정답 힌트**: 자바스크립트는 함수를 일급 객체로 다루므로, 부모의 상태를 변경하는 함수 자체를 자식 컴포넌트의 props로 주입하여 자식이 이벤트 발생 시 이를 콜백하도록 만듭니다.

### [ ] 5. 상태(State)가 변경되었을 때 화면이 갱신(Re-render)되는 물리적 과정을 설명할 수 있다.
* **스스로 답해보기**: React 컴포넌트는 자바스크립트의 일반 함수인데, `setState`를 호출하면 함수 내부 전체가 어떻게 다시 실행되어 브라우저 화면이 바뀌는지 그 라이프사이클을 설명해 보세요.
* **정답 힌트**: `setState`가 트리거되면 컴포넌트 함수가 재실행되며 새로운 가상 DOM(Virtual DOM)을 만들고, 이전 가상 DOM과 비교하여 바뀐 부분만 가려낸 뒤 실제 브라우저 DOM에 타깃 업데이트(Commit)를 수행합니다.

## 3. 비동기 처리 및 Effect

### [ ] 6. 최초 페이지 로딩 시 데이터를 가져올 때 `useEffect`를 사용해야만 하는 이유를 설명할 수 있다.
* **스스로 답해보기**: 컴포넌트 함수 내부(바깥쪽)에서 그냥 `fetch()`를 직접 호출하면 왜 무한 렌더링에 빠지거나 경고가 생길까요?
* **정답 힌트**: `useEffect` 없이 렌더링 스코프 안에서 직접 비동기 상태를 수정(`setState`)하면, 렌더링 -> 상태수정 -> 렌더링 -> 상태수정의 무한 리렌더링 지옥에 빠지게 됩니다.

### [ ] 7. 버튼 클릭으로 인한 데이터 저장/삭제 API를 `useEffect`가 아닌 이벤트 핸들러에서 호출해야 하는 이유를 안다.
* **스스로 답해보기**: 사용자가 의식하고 실행한 능동적 저장 작업은 왜 `useEffect` 의존성 배열 감지로 풀면 위험할까요?
* **정답 힌트**: `useEffect`는 컴포넌트의 상태가 화면과 동기화되어 나타나는 '부수 효과(Side Effect)'를 위한 것이며, 사용자의 명시적 조작은 버튼의 `onClick` 이벤트 안에서 직접 처리하는 것이 흐름을 명확하고 안전하게 만듭니다.

### [ ] 8. 의존성 배열(Dependency Array)의 의미와, 그 안에 값이 누락되었을 때 생기는 오류(Stale Closure 등)를 설명할 수 있다.
* **스스로 답해보기**: `useEffect` 내부에서 참조하는 외부 상태나 props가 있을 때 의존성 배열에 이를 넣지 않으면 어떤 현상이 일어나나요?
* **정답 힌트**: 의존성 배열에 참조 변수를 누락하면, React는 마운트 시점의 낡은 상태 값(Stale)을 가둔 렉시컬 스코프를 계속 참조하게 되어 최신 데이터 변경을 화면이 인지하지 못합니다.

### [ ] 9. 데이터 비동기 조회 시 왜 항상 로딩(`loading`), 오류(`error`), 데이터(`data`)의 3단 상태가 필요한지 타당성을 설명할 수 있다.
* **스스로 답해보기**: 네트워크 데이터가 즉각 오지 않을 때를 대비하지 않는다면 사용자 화면에는 어떤 어색한 현상이 유발되는지 설명하세요.
* **정답 힌트**: API 호출은 지연이 발생하므로 로딩 상태가 없으면 빈 화면이 정지된 상태로 보여 멈춘 것처럼 보이고, 에러 분기가 없으면 장애 상황에서 앱이 크래시(하얀 화면)를 겪게 됩니다.

## 4. 고급 비동기 동기화 및 구조화

### [ ] 10. `useEffect`의 Cleanup 함수가 필요한 이유와 디바운싱(Debounce)에서 작동하는 메커니즘을 설명할 수 있다.
* **스스로 답해보기**: 키보드를 연속으로 누르는 와중에 저장 타이머를 어떻게 효과적으로 차단하고 멈추었을 때 한 번만 전송하게 만드나요?
* **정답 힌트**: 의존성 변수가 변경될 때마다 다음 Effect 시작 전 Cleanup 함수가 먼저 동작하므로, 타이핑 중에는 이전 등록한 `setTimeout`을 매번 취소(Clear)해주어 타이핑을 멈춘 뒤에만 타이머가 만료되도록 설계할 수 있습니다.

### [ ] 11. API 통신 파일(`pageApi.ts`)과 UI 컴포넌트 파일을 물리적으로 분리했을 때 얻는 이점을 설명할 수 있다.
* **스스로 답해보기**: 컴포넌트 파일에 Axios 코드가 직접 섞여 있지 않음으로써 아키텍처 관점에서 무엇이 깨끗해지는가?
* **정답 힌트**: UI 컴포넌트는 화면 렌더링에만 집중하고, 데이터 수집 방식은 API 모듈에 격리하여, API 주소 변경이나 라이브러리 교체(Axios -> Fetch) 시에 UI 코드 변경 없이 데이터 레이어만 수정할 수 있게 해 줍니다.

### [ ] 12. React 개발 모드(StrictMode)에서 API 요청이 왜 2번씩 실행되며, 이것이 왜 정상인지 설명할 수 있다.
* **스스로 답해보기**: 실서비스 배포 시에도 API가 똑같이 2번 호출되나요? StrictMode가 이를 강제해 주는 의미는 무엇인가요?
* **정답 힌트**: StrictMode는 컴포넌트가 마운트 해제 후 재마운트되는 과정을 연달아 수행하여 개발 단계에서 CleanUp이 누락되었는지를 체크하는 도구이며, 실제 운영(Production) 빌드에서는 1번만 실행됩니다.
